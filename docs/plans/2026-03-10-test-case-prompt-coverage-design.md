# Test Case Prompt Coverage Design

**Date:** 2026-03-10

**Scope:** 仅针对普通需求生成测试用例的提示词链路，提升覆盖深度和 case 数量引导，不调整 API 用例生成链路。

## Problem

当前普通需求生成提示词已经要求“尽可能多覆盖需求场景”，但约束还不够硬：
- 覆盖维度只明确到正常流、边界流、异常流，非功能场景要求偏弱。
- `case_count` 更像参考值，不足以强制模型继续补足覆盖。
- 回归测试主要关注 JSON 合约和自检，没有锁定覆盖维度与数量导向。

## Goals

- 明确要求覆盖主流程、关键分支、边界条件、异常处理、空值与非法输入。
- 在提示词层面显式要求补充性能、兼容性、安全、稳定性等非功能场景。
- 将“生成更多 case”升级为硬性倾向：
  - `case_count > 0` 时，至少参考目标数量；如果覆盖维度仍不完整，需要继续补充。
  - `case_count <= 0` 时，不设上限，尽可能多生成。
- 用自动化测试锁定上述约束，避免后续回退。

## Non-Goals

- 不修改 API 接口测试用例生成提示词。
- 不新增生成后重试或数量下限校验逻辑。
- 不修改前端交互与入参协议。

## Design

### Prompt Changes

- 在 `test_case_generator.system_template` 中加入覆盖硬要求，要求模型必须覆盖：
  - 主流程
  - 关键分支
  - 边界条件
  - 异常处理
  - 空值与非法输入
  - 权限与安全
  - 性能与容量
  - 兼容性
  - 稳定性与恢复性
- 在 `test_case_generator.human_template` 中增加数量与拆分策略：
  - 同一功能点下，正常、异常、边界、性能、安全、兼容性、稳定性场景应拆分为不同 case。
  - `case_count` 为正时，用“至少参考目标 N 条，覆盖不足应继续补充”表述。
  - 保留现有 JSON 输出和自检约束。

### Prompt Assembly Changes

- 调整 `TestCaseGeneratorPrompt.format_messages()` 中 `quantity_instruction` 的文案生成逻辑，使其与新的覆盖策略一致。

### Regression Tests

- 为普通需求生成提示词增加覆盖断言：
  - 主流程、关键分支、边界条件、异常处理
  - 性能、兼容性、安全、稳定性
  - 至少参考目标数量
  - 覆盖不足继续补充
- 保留并复用已有 JSON 合约断言。

## Risks

- 提示词更长后，模型可能更容易输出冗余描述。
- 强化“更多 case”后，部分模型可能更容易返回过长响应。

## Mitigations

- 保持 JSON 输出硬约束不变。
- 用测试锁定覆盖指令，但不引入额外后处理逻辑，先观察提示词优化收益。
